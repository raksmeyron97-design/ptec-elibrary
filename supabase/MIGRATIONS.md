# Database Migrations Guide

This project uses [Supabase migrations](https://supabase.com/docs/guides/deployment/database-migrations) to version the Postgres schema. All migration files live in `supabase/migrations/` and are applied **sequentially by filename**.

## Hosted-database pipeline (CI) — added 2026-07-16

Migrations are applied to the hosted DB by `.github/workflows/migrate.yml`:

- **PRs** touching `supabase/migrations/**` get a `supabase db push --dry-run`
  against the hosted DB (shows what would apply). The `e2e` CI job separately
  proves the whole chain applies cleanly from scratch (`supabase start` on a
  fresh local stack).
- **Merges to main** touching `supabase/migrations/**` apply pending
  migrations for real. No more dashboard SQL-editor applies — that workflow
  left the history table empty and the code full of "pre-migration" fallbacks.

Setup (one secret): add `SUPABASE_DB_URL` in GitHub → Settings → Secrets →
Actions, set to the **Session pooler** connection string from the Supabase
dashboard (Connect → Session pooler, port 5432). Not the direct `db.<ref>`
host (IPv6-only — unreachable from GitHub runners) and not the transaction
pooler on 6543.

History bootstrap: everything through `0098` was applied by hand before this
pipeline existed, so the first workflow run executes
`scripts/migrations/bootstrap-history.mjs`, which marks the versions in
`supabase/applied-baseline-2026-07-16.txt` (verified live via object probes
on 2026-07-16) as applied without running any SQL. It is a no-op afterwards.

## Creating a new migration

```bash
supabase migration new <name>
```

This generates an empty file named `supabase/migrations/<timestamp>_<name>.sql` (e.g. `20260704101530_create_events_table.sql`). Write your SQL in that file, then apply it:

```bash
supabase db reset        # local: rebuild the local DB from all migrations
supabase db push         # hosted: apply pending migrations to the linked project
```

## Naming conventions

- **Timestamps are generated by the Supabase CLI** — never hand-write or reorder them. The 14-digit prefix (`YYYYMMDDHHMMSS`) determines apply order.
- Use **descriptive snake_case names** that say what the migration does:
  - ✅ `create_events_table`, `add_user_preferences`, `fix_reviews_rls`, `grant_anon_home_stats`
  - ❌ `update`, `fix`, `changes2`, `new_migration`
- One concern per migration. If a change needs a new enum value *and* code that uses it, split it into two migrations — Postgres requires enum values to be committed before use (see archived `0039`/`0040` for a real example).
- Prefer idempotent SQL where practical (`IF NOT EXISTS`, `CREATE OR REPLACE`, `DROP ... IF EXISTS` before `CREATE POLICY`), so a migration can be re-run safely during development.

## Current baseline (squashed schema)

`00000000000000_initial_schema.sql` is a **squashed baseline** containing the consolidated final state of the original migrations `0001`–`0052` (tables, indexes, views, functions, triggers, RLS policies, grants, and seed data). Its all-zero timestamp guarantees it sorts — and applies — before every other migration.

- A **fresh database** (e.g. `supabase db reset` locally) gets the entire schema from this one file, then applies anything newer (`0053_…`, `0054_…`, and future timestamped files) on top.
- **Existing databases** that already ran `0001`–`0052` must NOT re-run the baseline. If the CLI reports history drift, reconcile with `supabase migration repair` (mark the baseline as applied, and the archived versions as reverted).

## The `_archive/` folder

`supabase/migrations/_archive/` holds the original 52 migration files (`0001`–`0052`) for historical reference — useful for auditing when/why a change was made, or for reconciling an old environment. The Supabase CLI ignores subdirectories, so these files are never applied. **Do not move them back** into `migrations/`; the baseline already contains their final state.

## When to squash again

Squash the migration history when any of these hold:

- The folder accumulates **~50 migrations** since the last squash.
- A **major release** ships and all environments (local, staging, production) are confirmed to be on the same migration version.
- Migration replay on a fresh database becomes slow or fragile (e.g. old migrations reference since-dropped objects).

Squash procedure (summary):

1. Verify every environment has applied all migrations up to the squash point.
2. Produce a single file with the **final state** of the schema — resolve `CREATE OR REPLACE` chains, dropped/recreated policies, and `ALTER`ed columns into one definition each. Diffing against `supabase db dump --schema public` is the most reliable way to capture the true final state (it also catches dashboard-made changes that never got a migration file).
3. Name it so it sorts before all remaining migrations (or use the CLI-supported baseline flow).
4. Move the squashed originals into `_archive/`.
5. Test: `supabase db reset` on a fresh local stack must succeed, and the resulting schema must match production (`supabase db diff` should be empty).
6. Run `supabase migration repair` against hosted environments so migration history matches the new file list.

## Gotchas specific to this project

- **Dashboard drift**: some objects (the `user_role` enum, several `catalog_books` columns) were historically created via the Supabase dashboard, not migrations. Always create schema changes as migration files so `db reset` keeps working.
- `research_reports` is the DB name for what the UI calls **theses**; the permissions resource is `research`.
- Functions marked service-role-only (e.g. `increment_ai_usage`, `check_rate_limit`, `cleanup_rate_limit`) must keep their `REVOKE` statements when touched — they are part of the security model.
