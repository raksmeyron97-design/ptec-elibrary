# លំហាត់សាកល្បងសង្គ្រោះប្រព័ន្ធ (Tabletop Exercises)

_បានបង្កើតនៅថ្ងៃទី ១២ ខែកក្កដា ឆ្នាំ២០២៦។ នេះគឺជាសេណារីយ៉ូ (Scenarios) សម្រាប់យកមកសាកល្បងមើលថាតើសៀវភៅណែនាំ (`RUNBOOKS.md`) ពិតជាដំណើរការបានល្អដែរឬទេ នៅពេលមានបញ្ហាធំៗកើតឡើង។_

## របៀបធ្វើការសាកល្បង (How to run)

រៀងរាល់ ៦ ខែម្តង ក្រុមការងារវេប (Web-team) និងនាយកបណ្ណាល័យ គួរតែអង្គុយជុំគ្នា ហើយជ្រើសរើសយកសេណារីយ៉ូមួយមកពិភាក្សា៖ "បើរឿងនេះកើតឡើងមែន តើយើងត្រូវធ្វើអ្វីខ្លះកាលពី១ម៉ោងមុន? ឥឡូវហ្នឹង? និងនៅថ្ងៃស្អែក?" កត់ត្រាចំណុចខ្វះខាតដែលរកឃើញបញ្ចូលទៅក្នុង `OPERATIONS-AUDIT.md` (បញ្ជីហានិភ័យ)។

---

## TT-1: The Silent Dropper (ទិន្នន័យបាត់បង់ដោយស្ងាត់ៗ)

**ស្ថានភាព**: បណ្ណារក្សម្នាក់បានរាយការណ៍ថា សៀវភៅ ៥ ក្បាលដែលគាត់ទើបតែបញ្ញូលកាលពីសប្តាហ៍មុន ស្រាប់តែបាត់រូបក្របទាំងអស់ ហើយបើក PDF លែងចេញ (លោត Error 404)។ គាត់ប្រាកដចិត្តថាគាត់បាន Upload ជោគជ័យហើយកាលពីសប្តាហ៍មុន។

**អ្វីដែលត្រូវរំពឹង (Expectation)**:
1. ផ្ទៀងផ្ទាត់សិន: ចូល `admin/data-quality` តើមានឯកសារខូច (broken links) ច្រើនប៉ុណ្ណា? តើវាខូចតែ ៥ ក្បាលហ្នឹង ឬខូចទាំងអស់?
2. ឆែកមើលប្រវត្តិ: ចូល `admin_audit_log` ថាតើមាន Admin ណាផ្សេងលុបវាដោយចៃដន្យឬទេ? ពិនិត្យ `content_versions` ថាតើមានអ្នកកែ Metadata ដែរឬទេ?
3. សង្គ្រោះទិន្នន័យ (Recovery): បើឯកសារបាត់ពី ZimaOS មែន តើយើងអាចយកវាពីទីណាបានវិញ? (ចម្លើយ: ទាញយកពី Backup `/mnt/backup-disk/zima-mirror/` កាលពីថ្ងៃដែលឯកសារនៅល្អ)។

**មេរៀន (Follow-up)**: តើ Backup ឯកសារនៅលើ ZimaOS មានរត់ត្រឹមត្រូវរាល់យប់ដែរឬទេ? តើ Alert `file-snapshot-stale` ដើរឬអត់?

## TT-2: The Ransomware Hit (ការវាយប្រហារចាប់ជំរិត)

**ស្ថានភាព**: ម៉ាស៊ីន ZimaOS ត្រូវបានរងការវាយប្រហារដោយមេរោគជម្រិតទារប្រាក់ (Ransomware)។ ឯកសារ PDF ទាំងអស់ត្រូវបានវាចាក់សោរ (Encrypted) ក្លាយជាឯកសារអានមិនបាន។ ម៉ាស៊ីនមិនអាចប្រើការបានទៀតទេ។ វេបសាយនៅបើកចេញ ប៉ុន្តែចុចមើលឯកសារអត់បាន។

**អ្វីដែលត្រូវរំពឹង**:
1. ការឆ្លើយតបដំបូង: ផ្តាច់ម៉ាស៊ីន Zima នោះចេញពីអ៊ីនធឺណិត និងបណ្តាញផ្ទៃក្នុងភ្លាមៗ។
2. បន្តដំណើរការ (Triage): វេបសាយនៅដើរធម្មតា (ដោយសារ Vercel + Supabase) ចំណែកឯកសារទាញអត់ចេញទេ។ តើយើងអាចប្តូរ `ZIMA_API_URL` ទៅកន្លែងបណ្តោះអាសន្នបានទេ?
3. ការសង្គ្រោះ: តើ Backup ច្បាប់ចម្លងទី៣ របស់យើង (R2 bucket ដែលដាច់ដោយឡែក) មានសុវត្ថិភាពឬទេ? យើងត្រូវទិញ/ដំឡើង ZimaOS ថ្មី ហើយទាញយកឯកសារទាំងអស់ពី R2 មកវិញ។ ត្រូវចំណាយពេលប៉ុន្មាន?

**មេរៀន**: តើយើងមានការ Backup ឯកសារច្បាប់ចម្លងទី៣ ទៅកាន់ R2 bucket ដោយសុវត្ថិភាពហើយឬនៅ? (បច្ចុប្បន្ននៅមានចំណុចខ្វះខាតត្រង់នេះ - មើលហានិភ័យ R1)។

## TT-3: The Disgruntled Admin (Admin ខឹងសម្បារ)

**ស្ថានភាព**: អតីតបុគ្គលិកថ្នាក់ Admin ម្នាក់ដែលទើបតែត្រូវបញ្ឈប់ពីការងារ បានលួច Login ចូលមកក្នុងផ្ទាំង Admin ទាំងយប់ រួចចុច Archive និងកែឈ្មោះចំណងជើងសៀវភៅអស់ជាង ១០០០ ក្បាល។ ព្រឹកឡើង ទំព័រមុខវេបសាយបាត់សៀវភៅអស់ច្រើនណាស់។

**អ្វីដែលត្រូវរំពឹង**:
1. ការទប់ស្កាត់ (Contain): បិទគណនីនោះចោលភ្លាមៗ (RUNBOOKS §I8)។ ប្តូរ Password និងផ្តាច់ Session ចាស់ៗ។
2. វាយតម្លៃការខូចខាត: ចូលមើល `admin_audit_log` តើគាត់បានកែប្រែឯកសារណាខ្លះ ចាប់ពីម៉ោងប៉ុន្មានដល់ម៉ោងប៉ុន្មាន?
3. ការសង្គ្រោះ: ដោយសារគាត់គ្រាន់តែកែ Metadata យើងអាចប្រើ `content_versions` (ប្រវត្តិឯកសារ) ដើម្បី Restore សៀវភៅទាំង ១០០០ ក្បាលនោះមកសភាពដើមវិញ ដោយសរសេរ Script តូចមួយតាម API។

**មេរៀន**: តើនីតិវិធីនៅពេលបុគ្គលិកឈប់ពីការងារ (Offboarding) អនុវត្តបានតឹងរ៉ឹងឬទេ? (ម៉េចបានជាគាត់នៅមានសិទ្ធិចូល?) ត្រូវពិនិត្យឡើងវិញនូវ RUNBOOKS §M17។

## TT-4: The Traffic Tsunami (រលកនៃសំណើ)

**ស្ថានភាព**: នៅរដូវប្រឡង វេបសាយស្រាប់តែដើរយឺតខ្លាំង (Error 503 Service Unavailable)។ ឆែកមើលទៅឃើញមានសំណើ (Requests) ជាង ២ម៉ឺន ក្នុងមួយនាទី បាញ់ចូលមកទំព័រទាញយកឯកសារ។

**អ្វីដែលត្រូវរំពឹង**:
1. វិភាគ: ចូលទៅ Cloudflare ដើម្បីមើលថាតើ Traffic នេះមកពីស្រុកខ្មែរ (សិស្សពិតប្រាកដ) ឬមកពីបរទេស/IP ប្លែកៗ (Botnet/DDoS)?
2. កាត់បន្ថយសម្ពាធ (Mitigate): ចូល Cloudflare → បើក Under Attack Mode ជាបណ្តោះអាសន្ន។ បើកមុខងារ Rate Limit តឹងរ៉ឹងបន្ថែមទៀត (RUNBOOKS §I13)។
3. ស្ថិរភាព DB: ចូល Supabase Dashboard តើការភ្ជាប់ (Connections) ពេញឬទេ?

**មេរៀន**: តើ Caching របស់យើងលើ Vercel ដំណើរការបានល្អទេ? តើ Rate limiting ក្នុង DB រត់បានត្រឹមត្រូវឬទេ?

---

## TT-5: The Fat-Finger DROP (ភ្លាត់ដៃលុប Table)

**ស្ថានភាព**: ប្រធានក្រុមការងារវេប កំពុងព្យាយាម Run កូដសម្អាតទិន្នន័យ (Migration) ប៉ុន្តែបានសរសេរកូដខុស ធ្វើឲ្យប៉ះពាល់ដល់តារាង `books` (ឧ. `DELETE FROM books WHERE id IS NOT NULL`)។ ទិន្នន័យសៀវភៅជាង ៥០០០ ក្បាលត្រូវរលុបបាត់ពី Database។

**អ្វីដែលត្រូវរំពឹង**:
1. រក្សាសភាពស្ងប់ស្ងាត់: វេបសាយខូចបាត់ទៅហើយ។ ដាក់សញ្ញា "Under Maintenance" ។
2. ស្វែងរក Backup: ចូលទៅមើល Backup ចុងក្រោយបង្អស់ (ពីយប់មិញ `backup-db.mjs`)។
3. ការសង្គ្រោះ: Run script `restore-drill.mjs` ដើម្បីទាញយកទិន្នន័យក្នុង JSONL មកបញ្ចូលវិញ (RUNBOOKS §6.1/6.2)។ ទិន្នន័យថ្មីដែលបញ្ចូលក្រោយការ Backup យប់មិញ អាចនឹងបាត់បង់ខ្លះ (ត្រូវបញ្ចូលថ្មីឡើងវិញ)។

**មេរៀន**: នេះជាមូលហេតុដែលយើងមិនអនុញ្ញាតឲ្យរត់កូដ UPDATE/DELETE ផ្ទាល់លើ Database ដោយគ្មានការសាកល្បងច្បាស់លាស់។ ការតម្រូវឲ្យ Backup មុនធ្វើការ (RUNBOOKS §M6) គឺជារឿងដាច់ខាត។

## TT-6: The Silent Config Decay (ការផ្លាស់ប្តូរដោយលាក់កំបាំង)

**ស្ថានភាព**: ការជូនដំណឹង `backup-failed` បានឈប់លោតតាំងពី ៣ ខែមុនមកម៉្លេះ តែគ្មានអ្នកណាដឹង។ ថ្ងៃមួយ DB គាំងទើបដឹងថា Backup ចុងក្រោយគឺកាលពី ៣ ខែមុន។ ហេតុអ្វីទើបគ្មានអ្នកដឹង? ដោយសារតែមានគេកែប្រែកូដសម្ងាត់ `SMTP_PASSWORD` ដោយមិនបានប្រាប់នរណា។

**អ្វីដែលត្រូវរំពឹង**:
1. ការការពារ (Prevention): ដើម្បីការពាររឿងនេះ ទើបយើងមាន Probe ត្រួតពិនិត្យសុខភាព (`backupAgeHours`) ក្នុង `/api/health` ដែលឆែកពីខាងក្រៅ។ តើ Healthchecks.io កំពុងដំណើរការដែរឬទេ?
2. សវនកម្មប្រចាំខែ (Audit): នៅក្នុង M4 (RUNBOOKS) មានការតម្រូវឲ្យពិនិត្យ Config Fingerprint រាល់ត្រីមាស។

**មេរៀន**: ត្រូវប្រាកដថា Monitoring ដើរដោយស្វ័យប្រវត្តិ និងផ្តាច់ចេញពីប្រព័ន្ធខាងក្នុង (External checks) ព្រមទាំងការធ្វើតេស្តសាកល្បងរាល់ត្រីមាស (Drills) ត្រូវបានអនុវត្តដោយមិនខកខាន។
