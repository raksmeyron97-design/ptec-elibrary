# គោលការណ៍គ្រប់គ្រងទិន្នន័យ (Data Governance Policy)

_បានបង្កើតនៅថ្ងៃទី ១២ ខែកក្កដា ឆ្នាំ២០២៦ (ការងារទី ៦ ក្នុងផែនការ)។ ម្ចាស់ដែលអនុម័ត: នាយកបណ្ណាល័យ; អ្នកគ្រប់គ្រងបច្ចេកទេស: ប្រធានក្រុមការងារវេប។ ត្រួតពិនិត្យជារៀងរាល់ឆ្នាំ ឬក្រោយពេលមានឧប្បត្តិហេតុណាមួយពាក់ព័ន្ធ។ ជួរឈរ "ការអនុវត្តបច្ចេកទេស" បង្ហាញពីកូដ/migration ដែលអនុវត្តច្បាប់នីមួយៗ — គោលការណ៍ និងការអនុវត្តជាក់ស្តែង មិនត្រូវខុសគ្នាឡើយ។_

វិធានឯកជនភាពត្រូវបានអនុវត្តរួចជាស្រេចនៅក្នុងកូដ៖ មិនមានរក្សាទុក IP ពិតប្រាកដក្នុងបញ្ជីស្វែងរកទេ (ប្រើត្រឹម HMAC session hash ដែលប្តូររាល់ថ្ងៃប៉ុណ្ណោះ — `lib/search/analytics.ts`); ប្លុក Bots ចោលនៅពេលកត់ត្រា; ក្នុង Security log មិនដែលមាន Passwords, Tokens, Cookies ឬអត្ថបទសារឡើយ (`lib/security-log.ts`); តារាងវិភាគទិន្នន័យត្រូវបានចាក់សោ RLS-locked ជាសេវាកម្ម (0084/0087) ហើយអាចអានបានតែតាមរយៈសកម្មភាពដែលការពារដោយបណ្ណារក្សប៉ុណ្ណោះ។

## 3. ការលុបគណនី (Account deletion)

ពេលមានការស្នើសុំត្រឹមត្រូវពីអ្នកប្រើប្រាស់ (ឬតាមរយៈមុខងារលុបដោយខ្លួនឯង)៖
1. លុបគណនី Supabase auth → ទិន្នន័យក្នុង `profiles` និងអ្វីដែលជារបស់ User នោះនឹងលុបតាមស្វ័យប្រវត្តិ (កំណត់ចំណាំ, វឌ្ឍនភាពអាន, បញ្ជីសៀវភៅសន្សំ, ការចុះឈ្មោះទទួលសារ)។
2. ការវាយតម្លៃ/មតិ (Reviews/comments): លុបចោល, ឬលាក់ឈ្មោះ (anonymize) បើគិតថាសំខាន់សម្រាប់ការសន្ទនា។
3. សារទំនាក់ទំនង: រក្សាទុកតាមច្បាប់ទំនាក់ទំនង (legitimate interest) តែផ្តាច់ចេញពីគណនីនោះ។
4. កំណត់ត្រាសវនកម្ម (Audit-log) ដែលពាក់ព័ន្ធនឹង *សកម្មភាព Admin* របស់ User នោះ ត្រូវបានរក្សាទុក (ជាកំណត់ត្រាគណនេយ្យភាព) — ចំណុចនេះមានបញ្ជាក់ក្នុងអត្ថបទឯកជនភាព។
5. Backups: ការលុបនឹងបាត់ទៅវិញពេល Backup ចាស់ៗត្រូវលុបចោល (≤ ៦ ខែ); បើមានការ Restore Backup ចាស់ គឺត្រូវតែយកបញ្ជីដែលបានលុបរួចហើយ មកកាត់ចោលម្តងទៀត (រក្សាទុកក្នុង deletion log)។

## 4. ការស្នើសុំដកចេញ ការប្តឹងតវ៉ា និងការកែតម្រូវ (Takedown, complaints & corrections)

| ការស្នើសុំ (Request) | ឆានែល | រយៈពេល (SLA) | នីតិវិធី |
|---|---|---|---|
| បណ្តឹងរក្សាសិទ្ធិ (Copyright) | Form ទំនាក់ទំនង / អ៊ីមែលទៅបណ្ណាល័យ | ឆ្លើយតបក្នុង ៣ថ្ងៃធ្វើការ | បិទការបោះពុម្ពសិន (ប្តូរ status ទៅ archived ព្រោះអាចដោះវិញបាន) រង់ចាំការត្រួតពិនិត្យ; DIR ជាអ្នកសម្រេច; កត់ត្រាក្នុង Folder ឧប្បត្តិហេតុ |
| សុំដកឯកសារស្រាវជ្រាវ (ពីអ្នកនិពន្ធ/ស្ថាប័ន) | ដូចគ្នា | ដូចគ្នា | ផ្ទៀងផ្ទាត់អត្តសញ្ញាណអ្នកស្នើសុំ → ដាក់ចូល archive ឬរឹតត្បិត → កត់ត្រាមូលហេតុក្នុង `review_note` |
| រាយការណ៍ពីព័ត៌មានឯកសារខុស | Form ទំនាក់ទំនង | ១០ ថ្ងៃធ្វើការ | បណ្ណារក្សកែតម្រូវតាមរយៈទម្រង់ (មានកត់ត្រាជំនាន់); បើមានការមិនឯកភាព DIR ជាអ្នកសម្របសម្រួល; ជូនដំណឹងដល់អ្នករាយការណ៍ |
| សំណើទិន្នន័យផ្ទាល់ខ្លួន/ឯកជនភាព (សុំចូលមើល, សុំលុប) | អ៊ីមែល | ៣០ ថ្ងៃ | តាម §3; ត្រូវផ្ទៀងផ្ទាត់អត្តសញ្ញាណជាមុនសិន |

Archived មិនស្មើនឹងការលុបចោល (Deleted) ទេ៖ ឯកសារក្នុង Archived នៅតែរក្សាប្រវត្តិរបស់វា ហើយអាចទាញមកវិញបាន (status workflow); ការលុបចោលទាំងស្រុង (True deletion) គឺប្រើតែនៅពេលមានការតម្រូវផ្នែកច្បាប់ប៉ុណ្ណោះ ហើយត្រូវតែកត់ត្រាក្នុង audit-log ជានិច្ច។

## 5. ធនធានសាធារណៈ ធៀបនឹង ឯកជន (Public vs private resources)

- សាធារណៈ = `status='published'` (ហើយចំពោះឯកសារ ត្រូវឆ្លងកាត់ proxy ទាញយកដែលមានការកត់ត្រា)។ អ្វីផ្សេងពីនេះ គឺអាចមើលបានតែក្នុងផ្ទាំង Admin ប៉ុណ្ណោះ។
- ប្រព័ន្ធបញ្ចេញទិន្នន័យផ្លូវការ (OAI-PMH, `/api/export`) តម្រូវឱ្យមានការផ្ទៀងផ្ទាត់ (Verification) ពីបណ្ណារក្សបន្ថែមទៀត — សូមមើល `METADATA-EXPORTS.md` §gating ។
- OAI ទាមទារឱ្យមានអាជ្ញាប័ណ្ណបើកចំហ (Open license) ផងដែរ។
- RLS គឺជាជញ្ជាំងការពារចុងក្រោយ៖ ច្បាប់គ្រប់គ្រងរបស់ Table នីមួយៗត្រូវបានបញ្ជាក់ក្នុង `RLS-MATRIX.md` និងធ្វើតេស្តដោយ `lib/rls.test.ts` ។

## 6. ការត្រួតពិនិត្យសិទ្ធិចូលប្រើ (Access review)

រៀងរាល់ត្រីមាស (RUNBOOKS §M12)៖ គ្រប់គណនី staff+ ត្រូវតែមានហេតុផលត្រឹមត្រូវ, ប្រើ MFA, បុគ្គលិកដែលឈប់ធ្វើការត្រូវដកសិទ្ធិចេញក្នុងថ្ងៃនោះតែម្តង (§M17), DIR ចុះហត្ថលេខាឯកភាព រួចទុកក្នុង shared drive។ `admin_audit_log` + `content_versions` ជួយបញ្ជាក់រាល់ការកែប្រែដោយមានអំណាច (Privileged change) ដែលកើតឡើងចន្លោះពេលត្រួតពិនិត្យ។

## 7. កាតព្វកិច្ចធ្វើឲ្យស្របគ្នា (Alignment duty)

រាល់ការផ្លាស់ប្តូរណាមួយចំពោះអ្វីដែលវេបសាយប្រមូល រក្សាទុក ឬបញ្ចេញ ត្រូវតែធ្វើបច្ចុប្បន្នភាពក្នុងពេលតែមួយនូវ៖ ឯកសារនេះ (DATA-GOVERNANCE.md), អត្ថបទឯកជនភាពសាធារណៈ, និងកូដអនុវត្ត (Migration/Cron)។ អ្នកត្រួតពិនិត្យកូដ (Reviewer) ត្រូវឆែកមើលចំណុចទាំងបីនេះឱ្យបានហ្មត់ចត់។
